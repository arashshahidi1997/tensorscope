# Stale ports in make dev-ui over remote SSH

## Summary
- `make dev-ui` becomes stale after VS Code Remote SSH disconnects/reconnects, requiring full reconnect to recover ports
- Root cause: single-shell recipe with background process + trap fails to clean up on unclean session drop; Vite silently drifts to a different port; VS Code port forwarding state goes stale

## Context
- The original Makefile recipe ran backend (`&`) and frontend in one shell with `trap` cleanup
- Over remote SSH, when the terminal session dies, `trap` doesn't fire → orphan processes hold ports
- Vite picks a new port silently, VS Code still forwards the old one → appears "stale"

## Resolution
- Replaced single-shell recipe with `screen`-based detached sessions
- Added `kill-ui` target: kills stale screen sessions + frees ports via `fuser`
- Split into `dev-api` / `dev-web` targets in separate screen sessions
- `dev-ui` orchestrates: kill stale → start both → curl-probe ports (triggers VS Code auto-forward)
- Added `logs-api` / `logs-web` to reattach, `ui-status` to check port health
- Vite now uses `--strictPort` to fail loudly instead of drifting
- Bound to `127.0.0.1` for clean SSH tunnel forwarding
- Print `http://` URLs for clickability in terminal
