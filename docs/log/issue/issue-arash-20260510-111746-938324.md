TensorScope Audit — multidimensional electrophysiology viewer
1. Executive summary
TensorScope is a working, end-to-end viewer with a real xarray backend, an Arrow-IPC wire, a React/Zustand/uPlot frontend, and useful slice/PSD/spectrogram views over (time, AP, ML) data. Most of the engineering is solid: server-side slicing is centralised in one function, the DAG/transform model exists, processing is cached, and the spectrogram_live path has been carefully tuned for performance. However, the data contract is built around three specific dim layouts ({time, AP, ML}, {time, channel}, plus their freq variants) and that assumption is welded into the server's view registry, the slicing function, the selection DTO, the wire format, and the React stores. The system will not gracefully handle (trial, time, channel), (unit, time), (subject, session, time), peri-event epochs, or any generic xarray DataArray. Several places also degrade scientific fidelity in subtle ways: timeseries display amplitudes are silently z-scored and offset on the server, the timeseries view caps at 32 channels with no UI signal, min/max downsampling places extrema at bucket-edge timestamps (not their true location), AP/ML rank indices replace real coord values in the spatial extractor, and the Arrow long-format encoding inflates payload size by O(n_dims × n_cells). My recommendation is redesign the wire contract first (see §7), keep the core stores and views, and refactor in place around a labeled-tensor envelope. A full rewrite is not warranted.

4. Frontend critique
State & selection. store/selectionStore.ts is well-structured (cursor vs. window vs. spatial vs. event are separate sub-objects), but it commingles transient hover state (hoveredId) with committed selection. Any consumer that pulls the whole store re-renders on hover. Fix: split into useSelectionStore (committed) and useHoverStore (transient). The viewportDuration field shadows the persistent Viewport model on the server — two concepts of "what's visible" with different lifetimes.

Data fetching. api/queries.ts is the right abstraction for cache+dedup, but it has no cancellation (F10) and no per-view rate-limiting. The makeNavigatorRequest workaround at queries.ts:229-248 pins selection to zeros to stabilise the React Query key — the actual cause is using the full request body as the key. A reduced cache key (name, view_type, time_range, max_points, downsample, selection.{ap,ml,freq,time} only when the view consumes it) would be cleaner.

Type safety. api/types.ts is hand-maintained and already drifting (F8). The Pydantic models have validation (ge=0.0, gt=0.0, extra="forbid") that the TS types don't reflect. Generate TS from the OpenAPI schema (/openapi.json is already exposed) — openapi-typescript is one command.

Decoding. api/arrow.ts has both a fast columnar path and a row-based legacy path. The fast path's coord-aware extractors (e.g. extractSpatialCells, arrow.ts:202-238) inappropriately rank-normalise AP/ML (F7). The 4-D spectrogram path was correctly identified as a perf issue and rewritten (arrow.ts:438-518) — but the same row-explosion problem still exists at the wire layer (F6).

View components. TimeseriesSliceView.tsx is the most complex view (~770 lines) and well-engineered for resize/gesture handling. But it relies on the server pre-z-scoring and offsetting (F3), so the frontend has no understanding of true amplitudes — ChartToolbar's "gain" knob multiplies an already-z-scored value. The 32-channel cap (F4) is invisible to the user. PSDHeatmapView (line 27-33) parses channel labels back out of strings ("AP{val}_ML{val}") — string-encoded coords on the wire is a smell.

Render-path work. WorkspaceMain.tsx:445-447 re-decodes the PSD-live payload and reruns extractors on every render of WorkspaceMain — and WorkspaceMain re-renders on every selection-store change (including hover, F23). Wrap in useMemo keyed on psdLiveQuery.data?.payload.

No worker offload. Despite api/dataSource.ts:179-182 mentioning a future worker, all decode/extract is on the main thread. For a 256-ch × 60 s spectrogram, this is the dominant interactive latency.

5. Backend critique
Slicing semantics are mixed (F11). apply_slice_request uses .sel(time=slice(...)) for time/freq (label-based) and .isel(AP=slice(int, int+1)) for AP/ML/channel (positional). Two different worlds in the same request DTO. If AP coords are physical mm positions (e.g. [0.4, 1.6, 2.8, ...]), ap_range: [1, 3] selects positional indices 1..3, not millimetres. There's no field naming convention to signal which is which.

Hardcoded dim assumptions (F1, F9). _VIEW_REGISTRY is exhaustive enumeration of dim sets — there's no compositional rule "this view needs time, this view needs freq, this view aggregates over the rest." The psd_live collapse at state.py:431 and spectrogram_live at state.py:530 both write non_time_dims = [d for d in sliced.dims if d != "time"] and flatten everything else into a "channel" axis. For (trial, time, channel) this conflates trials with channels in the PSD compute — a major scientific error.

Server-side display normalisation (F3). zscore_offset is applied on the wire path for every timeseries/navigator view. This belongs in the frontend (or as an opt-in transform in the DAG). Currently the wire response carries z-scored, vertically-offset values, but its meta.units (line 266) is whatever the original tensor's attrs["units"] was — i.e. the false unit. A user hovering a value on the timeseries cannot trust the y-axis at all.

Encoding cost (F6). encode_arrow_payload (state.py:919-962) builds an N-D index meshgrid and writes one row per cell with full coord values per row. This is xarray's .to_series().reset_index() style — readable, but at the wire level a (t=200, freq=64, AP=16, ML=16) ≈ 3.27M cells × 5 columns payload is ~120 MB pre-compression for what is fundamentally a 26 MB cube. Arrow IPC's compression mitigates but doesn't fix the fundamental shape mismatch. The right wire shape is one Arrow array per dimension coord + one N-D values array in a single record batch (shape encoded in schema metadata) — order-of-magnitude smaller and simpler to decode.

Min/max correctness (F5). At state.py:894-899, each bucket emits (time_vals[s], chunk.min(axis=0)) and (time_vals[e-1], chunk.max(axis=0)). The min/max values are correct per channel, but their assigned timestamps are bucket boundaries, not the actual argmin/argmax indices. A 1-ms spike near the centre of a 100-ms bucket gets rendered at the bucket edge — a 50-ms position error. For LFP/EEG visualisation that's tolerable; for unit/spike-level data it's misleading. Fix: per-feature argmin/argmax, then look up time_vals[start + argmin]. This costs ~2× memory but is still vectorised.

Processing pipeline (F14). apply_processing hardcodes the order CMR→bandpass→notch→spatial_median→zscore. Real workflows often need notch before CMR (when line noise is common-mode), or spatial median before bandpass. There's also a fallback at state.py:242-249 where any processing exception silently returns the raw data, with only a logging.warning — the API returns success, the client thinks processing was applied. This is a scientific safety hazard (F21). Should return meta.processing.applied: bool and surface the failure.

Cache (F18). When processing params change, _processed_cache.clear() runs synchronously inside set_processing, then the next slice request triggers apply_processing(node.data, ...) on the full tensor in the request handler. For a 1-hour 256-ch recording at 1.25 kHz (~4.6 GB float64 in memory), this is many seconds of synchronous work blocking a request. Move to background thread + an in-progress sentinel.

fs inference (F13). Both state.py:417 and state.py:707-713 infer fs from np.median(np.diff(time_vals)) (or the first 101 samples). Brittle if recording has gaps/jitter; preferable to require attrs["fs"] and only infer with a loud warning.

6. API contract critique
Single envelope for every view. TensorSliceRequestDTO (models.py:249-289) carries time_range, freq_range, channels, ap_range, ml_range, frame_time, max_points, downsample, psd_params, spectrogram_live_params — every view-specific field stays optional and gets validated by a custom validator. New views need DTO surgery + validator updates + frontend type mirroring. Better: a discriminated union per view ({view_type: "psd_live", params: PsdLiveParams, ...}) so each view's request shape is locally typed.

Selection is hardwired to a 5-axis ontology (F2). SelectionDTO.{time,freq,ap,ml,channel} works for the demo dataset, but the time field's ge=0.0 rules out negative-lag peri-event views — despite the registered perievent_epochs transform producing lag ∈ [-Δ, +Δ]. There's no DTO field for trial, epoch, event_id, unit, subject, session, band. The right shape is a dict-of-coords keyed on dim name: {coord: {time: 12.3, freq: 8.0, AP: 4, trial: 17}} with the server validating against the active tensor's coord schema.

Wire format confusion. Slice responses carry the data in payload: base64(Arrow IPC) (models.py:299-301) but the slice metadata is in JSON meta. Coord summaries on /tensors/{name} are JSON (models.py:92-100) but coord values in slices are stored as columns in the Arrow table. There are three encodings of "what is a coord array": JSON inline (CoordSummaryDTO.values), JSON sparse (min/max only), and Arrow column. Pick one.

No content negotiation. Every slice goes over JSON-wrapped, base64-encoded Arrow. Browsers and pyarrow both speak Arrow IPC natively — Content-Type: application/vnd.apache.arrow.stream would skip the base64 round-trip (~33% bandwidth waste).

No version field on the wire. Pipeline DTOs have version: "1.0" (models.py:571); TensorSliceDTO and SelectionDTO don't. Means future contract changes will silently break existing clients.

Type drift (F8). Hand-maintained TS types miss enabled, K, detrend. extra="forbid" on the server catches frontend typos server-side, but typos slipping through Partial<T> builders are a runtime hazard. Generate TS from OpenAPI.

7. Minimal target architecture
1. Labeled-tensor wire format. Replace long-format columnar Arrow with one Arrow record batch per slice carrying:

A data field: the full N-D values array (Arrow's FixedSizeList/Tensor extension or one column per leading-dim slice — there are several reasonable choices).
One Arrow array per dim, named by dim, holding the coord values for that slice.
Schema metadata: dims: [...], dtype, units, attrs: {...}, processing_applied: {...}, slice_provenance: {...}.
This kills F6 outright, makes F11 and F7 fixable in one place, and gives the frontend a real "labeled tensor" to render against instead of reconstructing semantics from string-encoded channel labels.

2. Generic dim-aware selection. Replace SelectionDTO with {coord: {<dim>: <value>}, ranges: {<dim>: [lo, hi]}, indexers: {<dim>: <int>}}. The server validates against the active tensor's actual coord schema. Selection persists per-tensor (or with a documented promotion rule across tensors with shared dims).

3. Compositional view registry. Replace _VIEW_REGISTRY: dict[frozenset[str], list[str]] with per-view requirements: {name, required_dims: [...], aggregates_over: [...], produces_dims: [...]}. available_views(tensor) becomes "all views whose required_dims ⊆ tensor.dims". Adding a new dim type doesn't require re-keying the registry.

4. Generic min/max with argmin/argmax (F5 fix). Per-bucket per-feature argmin/argmax to recover true extrema timestamps; emit (time_vals[s+argmin], val) pairs. Memory: 2× the current min/max arrays. Works for (time, *) regardless of trailing dims.

5. Wire-side view options decoupled from data. ProcessingParamsDTO becomes a per-tensor DAG node, not a session-wide blob. Z-score-and-offset becomes a frontend display option layered over true-units data, not a server-side transform.

6. Code-gen contracts. OpenAPI → TS via openapi-typescript in CI; CI fail on drift.

7. Worker-offloaded decode. A dedicated arrow.worker.ts decodes payloads off the main thread; views receive typed arrays.

8. Cancellation. Pass AbortSignal from React Query into fetch; pan/zoom drops in-flight requests automatically.

## Addendum — Style, axes & rendering (2026-05-10, follow-up pass)

This addendum responds to two user-reported issues and adds related findings discovered while tracing them.

### A1. Reported: log-scale button does not update Y-axis labels in PSD

**What I traced.** Three views host a "log" toggle: [PSDHeatmapView](frontend/src/components/views/PSDHeatmapView.tsx#L277-L284), [PSDCurveView](frontend/src/components/views/PSDCurveView.tsx#L194-L201), and [SpectrogramView](frontend/src/components/views/SpectrogramView.tsx#L165-L172). All three call `toggleFreqLogScale` on the shared `useAppStore` ([store/appStore.ts:108-112](frontend/src/store/appStore.ts#L108-L112)) and pass `freqLogScale` into `<YTicks logScale={freqLogScale && lo > 0} />` ([AxisTicks.tsx:65-86](frontend/src/components/views/AxisTicks.tsx#L65-L86)). Tick generation switches between `makeTicks` (linear) and `makeLogTicks` (log) — both pure functions. There is **no obvious code-level reason the labels should not update**: the props flow through, the `<span>` keys differ between linear and log outputs, React re-reconciles.

**Most likely root cause (priority order):**

1. **Narrow Y-tick column hides the log labels.** The grid template at [styles.css:1090-1099](frontend/src/styles.css) gives `.axis-y-ticks` a fixed **30px** column. With `transform: translateY(50%)` and `right: 2px`, log labels like `"1"`, `"2"`, `"5"`, `"10"` fit, but they may sit *behind* the canvas on the right edge or overlap each other when the freq span is < 1 decade (e.g. `[1, 30]` Hz with `makeLogTicks` produces 5 ticks crammed into the same vertical column). At small canvas heights (PSD curve panel is often ~120 px tall in default layouts), neighbouring spans overlap so heavily the log labels look identical to the linear ones.

2. **`freqLogScale && lo > 0` silently degrades to linear when the lower bound is 0.** [PsdParamsDTO.fmin](src/tensorscope/server/models.py#L236-L240) defaults to `1.0` so this normally doesn't bite, but if a user sets `fmin=0` (e.g. via pipeline import), the heatmap/curve/spec view all silently render linear ticks even though the canvas can still draw log values for the > 0 portion. The toggle visually does nothing in that state.

3. **Content really is the same in some narrow cases.** `makeLogTicks(1, 30)` = `{1, 2, 5, 10, 20}`; `makeTicks(1, 30, 4)` = `{10, 20, 30}`. Different sets — but a user looking at "10, 20" on both could plausibly miss that "1, 2, 5" appeared.

**Recommendation.** Verify in DevTools whether `<span>` text content actually changes when toggling. If yes, fix is CSS-side (widen column from `30px` to `auto`/`min-content`, or add a `dpr`-aware label-collision pruner). If no, log-mode tick generation has a state bug I haven't reproduced — most likely cause #2 above. Either way, **drop the `&& lo > 0` guard from the *labels* and only apply it to the *positions*** so the user always sees clearly distinct linear vs. log tick *values* even when the lower bound is non-positive.

**Related axis findings discovered while tracing this:**

- **A1a — `freqLogScale` is global, not per-view.** All three views toggle the same store flag ([appStore.ts:38](frontend/src/store/appStore.ts#L38)). A user who wants log on PSD curve but linear on the spectrogram cannot have it. Severity: low; UX papercut.
- **A1b — PSDCurveView ticks track full data range, not viewport.** [PSDCurveView.tsx:205](frontend/src/components/views/PSDCurveView.tsx#L205) passes `lo={fMin} hi={fMax}` from `data.freqs[0]/[end]`, not from any viewport state (the curve has no zoom hook). If you later add zoom, ticks won't follow. Severity: low (latent).
- **A1c — `makeLogTicks` returns no ticks when `lo <= 0`.** [AxisTicks.tsx:40](frontend/src/components/views/AxisTicks.tsx#L40). The canvas in PSDHeatmapView/SpectrogramView falls back to linear in the same condition, so they stay consistent — but the empty-tick edge case means the user sees ticks vanish entirely if zoom drops the lower edge to ≤ 0. Recommendation: clip `lo` to `max(lo, 1e-3)` for tick purposes; keep canvas bounds as-is.
- **A1d — `<XTicks>` on PSDCurveView and PSDSpatialView is rendered as an empty div.** [PSDCurveView.tsx:209](frontend/src/components/views/PSDCurveView.tsx#L209), [PSDSpatialView.tsx:127](frontend/src/components/views/PSDSpatialView.tsx#L127) — `<div className="axis-x-ticks" />` with no content. The X-axis label "Power (log₁₀)" sits next to no numerical scale; users cannot read absolute power values. Severity: medium (scientific readability).
- **A1e — Spectrogram X-axis (time) has no log option.** Linear-only `<XTicks>` at [SpectrogramView.tsx:216](frontend/src/components/views/SpectrogramView.tsx#L216). For peri-event / lag-axis spectrograms (which a future `(event, lag, freq)` tensor will produce), this is a missing knob.
- **A1f — Tick label format `v.toFixed(0)` rounds sub-Hz freqs to "0".** [AxisTicks.tsx:24, 57](frontend/src/components/views/AxisTicks.tsx#L24). With `niceStep=0.5`, a tick at `0.5 Hz` renders as `"0"` — silently identical to a tick at `0` or `0.4`. Affects sleep-band PSD where `[0.5, 4]` Hz delta is the principal range.
- **A1g — Y-tick label clipping at the edges.** With `bottom: 0%` + `transform: translateY(50%)`, the lowest tick label sits half *below* the canvas frame; the highest tick (`bottom: 100%`) sits half *above*. On dark backgrounds inside flex containers, the bottom label often gets clipped by the parent's overflow. Use `transform: translateY(50%)` together with a `padding` reservation on `.axis-y-ticks`, or switch to `top: calc(100% - ${pct}%)` + `translateY(-50%)` so labels stay inside the column.

### A2. Reported: spatial map shows pixel borders; prefer matplotlib smooth + jet

**What I traced.** [ChannelGridRenderer.ts](frontend/src/components/views/ChannelGridRenderer.ts) renders one `ctx.fillRect` per `(AP, ML)` cell with `GAP = 1` (line 56). That 1-pixel gap is the visible border between cells. Two layers stop a smooth/matplotlib feel:

1. **Discrete cells, no interpolation.** Even if `GAP=0`, each cell remains a sharp rectangle. Matplotlib's `imshow` defaults to `interpolation="antialiased"` (since 3.5; previously `"nearest"`) and most neuroscience figures use `interpolation="bilinear"` or `"bicubic"`, giving the smoothness the user is asking for.
2. **HSL-rotation colormap, not jet.** [ChannelGridRenderer.ts:8-12](frontend/src/components/views/ChannelGridRenderer.ts#L8-L12) implements `sequentialColor(t)` as `hsl(220 - t*160, 70%, ...)` — blue → green via HSL hue rotation. This is **not** jet; it's a homegrown ramp that's also not perceptually uniform. PSDSpatialView, SpatialMapSliceView, and the propagation view all share this renderer, so all three look "wrong" by the user's stated aesthetic.

**Recommendation (concrete fix path):**

1. **Switch the renderer to `ImageData` at native grid resolution + scaled blit.** Allocate `ctx.createImageData(nML, nAP)`, write `(r,g,b,255)` per cell, then `ctx.imageSmoothingEnabled = true; ctx.imageSmoothingQuality = "high"; ctx.drawImage(offscreen, 0, 0, canvasW, canvasH)`. This is faster than per-cell `fillRect`, gives bilinear smoothing for free, and removes the gap entirely. SpectrogramView already uses `createImageData` ([SpectrogramView.tsx:106](frontend/src/components/views/SpectrogramView.tsx#L106)) — same idiom.
2. **Replace `sequentialColor` with a proper colormap LUT.** Add a `colormaps.ts` with `jet`, `viridis`, `inferno`, `cividis` as 256-entry `Uint8Array` LUTs (or piecewise-linear functions). Make the colormap a per-view (or global) setting. **Caveat: jet is not perceptually uniform** (compresses blue/green, exaggerates red→cyan transitions, hides ordinal differences in the middle of the range). If the user wants the *matplotlib look*, viridis is the modern matplotlib default. Implement jet because it was requested, but I'd recommend defaulting to viridis with jet as an option.
3. **Drop `GAP` to 0** unless the user explicitly toggles a "show grid" overlay.
4. **Also fix the lookalike colormap in PSDHeatmapView and SpectrogramView.** Both define `infernoColor` / `valueToColor` as the same hand-rolled three-line approximation ([PSDHeatmapView.tsx:36-42](frontend/src/components/views/PSDHeatmapView.tsx#L36-L42), [SpectrogramView.tsx:11-18](frontend/src/components/views/SpectrogramView.tsx#L11-L18)). These are *not* matplotlib's inferno — they bear only superficial resemblance. Once the LUT module exists, point all three views at it.
5. **Add a colorbar.** None of the heatmap views show a min/max color scale. Users see colors but cannot read values. A 30-px-wide vertical colorbar with 3–4 tick labels alongside each heatmap closes the loop.
