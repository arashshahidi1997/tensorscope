**Severity:** medium — current propagation view is technically functional but unusable for spotting spatio-temporal patterns at workflow speed.
**Status:** open.

## Symptom

The propagation_frame view fetches a single `(AP, ML)` frame per slice request, keyed on `frame_time`. Stepping through frames means a fresh server round-trip for each, with the slice itself taking on the order of 10–30 ms but the request/response/decode/render pipeline adding enough overhead that the view is too slow to "play back" smoothly. The user can't scrub through a few seconds of cortical activity to spot waves / sources.

## Why this matters for the workflow

Sleep-event audit and propagation analysis (slow-wave fronts, K-complex spreading, ripple co-occurrence, sharp-wave-ripple dipole patterns) all hinge on watching the spatial map evolve over a small time window at smooth playback rates. Holoviews/HoloMap nailed this UX in cogpy v2.x by preloading every frame in a window into a list, indexing into it client-side, and animating with RAF. We need an equivalent.

## Suggested implementation

A small frontend-only feature, with a tiny server addition.

**Server:** add a slice variant that returns N frames in one round-trip:

```
POST /tensors/{name}/slice
  view_type: "propagation_movie"
  selection: {...}
  time_range: [t_lo, t_hi]
  n_frames: 60         # default ~ window_s * 30
```

Returns `(time, AP, ML)` shaped Arrow IPC payload. The slice path already does `xr.sel(time=slice(t_lo, t_hi))`; we just stop at the first time-axis-collapsing step and return the windowed cube with `n_frames` evenly spaced indices via the existing `downsample_time_axis` (lttb mode is fine for spatial-map use).

**Frontend:** new `PropagationMovieView` (or extend `PropagationController`):
- Single slice fetch on mount / window change.
- Decode once into a `Float32Array[]` of length `n_frames` (one heatmap buffer per frame).
- Playback state: `{frameIdx, isPlaying, fps}` in component-local state.
- RAF loop advances `frameIdx` at `fps` (default 24, slider 1–60). Wraps at `n_frames`.
- Timeline scrubber below the canvas (NOT bound to the global cursor — propagation playback is local; clicking commits the selected frame's time as the cursor via `onCommitSelection`).
- Render pipeline: reuse `ChannelGridRenderer.render()` with the current frame's buffer.
- Loading indicator while the bundle is in flight (covered by the placeholder pattern shipped in `5908b3e`).

**Why this is cheap:** 60 frames × 256 cells × 4 bytes = 60 kB per movie. The compute is just an `xr.sel(time=slice(...)).isel(time=evenly-spaced)` — sub-100 ms server-side. The only "real" work is keeping the canvas RAF loop responsive.

## Repro

1. Activate `propagation_frame` view on the sub-01/ses-04 LFP tensor.
2. Try to scrub time within e.g. a 2 s NREM bout — each step refetches a single frame; visible stutter and lag.
3. Compare to cogpy v2.x HoloMap rendering of the same window: smooth playback at 30 fps with no per-frame round-trip.

## Out of scope

- Persistent caching of preloaded movies across viewport changes (nice-to-have but TanStack Query already handles it for free if we key by `(name, time_range, n_frames)`).
- Color-scale auto-fitting per frame vs per movie (default to per-movie min/max for stable colour mapping during playback).
- 3-D propagation (interpolated across time) — orthogonal feature, defer.
