# G9 — validation export

## Why

From the spindle-review critique
(`docs/log/issue/issue-arash-20260511-182119-502518.md` G9): "After 4
hours of review there's nothing on disk." TensorScope's pipeline export
covers transforms; nothing covers human review decisions. The
event-review store (G3, shipped) holds the decisions in `localStorage`
— G9 makes them durable.

## What to build

A server endpoint that accepts a batch of decisions and writes them to
a parquet/csv next to the dataset. A frontend "Export" button drives it.

### Server

- New router `src/tensorscope/server/routers/event_review.py`.
- `POST /api/v1/events/{stream_name}/decisions`:
  - Body: list of `{event_id, status, decided_at, notes?, tags?}`
  - Writes to a stable path: `<dataset_dir>/review/<stream>__decisions.parquet`
  - File format: parquet preferred (round-trippable, typed). Fallback
    to csv if pyarrow isn't writable for some reason.
  - Atomic write: write to `.tmp` then rename.
- `GET /api/v1/events/{stream_name}/decisions`:
  - Returns the previously-saved decisions so a fresh browser session
    can pre-populate the store.
- The bundled audit dataset (`/tmp/pixecog_tensorscope_spindle_audit_*/`)
  is writable, so the export lands next to the source files. For
  read-only bundles, return a clear 403 with a message pointing at
  `--write-dir` (a CLI option to add later if needed).

### Frontend

- New API client methods: `exportEventDecisions(streamName, body)` and
  `importEventDecisions(streamName)`.
- "Export decisions" button in the event panel header. Pulls all
  decisions for `(tensor, streamName)` from the store, posts to the
  server.
- On app boot: if the server has decisions for the active stream and
  the local store is empty (or stale), offer to seed the store from
  the server.
- A small status row: "Last saved: 2 min ago · 312 decisions on disk".

### DTOs

- `EventDecisionDTO` mirrors `EventDecision` in
  `frontend/src/store/eventReviewStore.ts`.
- `EventDecisionBatchDTO`: `{decisions: list[EventDecisionDTO]}`.

### Tests

- Backend: round-trip — POST a batch, GET it back, assert equality
  (including notes/tags).
- Atomic-write test: simulate a crash mid-write, verify the existing
  decisions file isn't corrupted.
- Frontend: API client smoke tests; store-rehydration logic on import.

## Acceptance

1. Click "Export decisions" → server writes
   `<dataset_dir>/review/spindle_ieeg_cortex_blobs__decisions.parquet`.
2. The parquet has one row per decided event; columns: `event_id`,
   `status`, `decided_at`, `notes`, `tags`.
3. Reload the browser → status row reads from disk.
4. Subsequent edits + re-export round-trip cleanly.

## Out of scope

- Multi-reviewer / merge semantics — single-user only in v0.
- A "decisions browser" that lists decisions across sessions / datasets.
- Inter-rater agreement scoring — defer.
- Integration with downstream pipelines (e.g. a `loadValidated()`
  helper for the user's downstream analysis scripts). Path of the
  output is stable so they can read it directly.
