# G5 — detector comparison overlay

## Why

From the spindle-review critique
(`docs/log/issue/issue-arash-20260511-182119-502518.md` G5): when the
spindle detector and ripple detector fire within ~100 ms on the same
channel, that's usually broad-band noise / artifact. TensorScope's
event panel shows ONE event stream at a time. There's no way to overlay
two and highlight coincidences. Reviewers need this to spot detector
crosstalk fast.

## What to build

The event panel + timeseries event markers gain multi-stream support.
A reviewer can pin 2+ streams; coincidences within a configurable
window light up.

### State

- New Zustand store slice (probably extend `selectionStore` or a small
  `eventStreamsStore`): `pinnedStreams: string[]`, `coincidenceWindow:
  number` (seconds, default 0.1), `coincidenceColor: string`.
- Default: only the first stream is "pinned" (matches current
  behavior). User can add more from a dropdown.

### Event table

- Tabs or a small "Streams: [spindle][ripple][+add]" strip at the top
  of the events tab — clicking toggles which stream is active in the
  table view.
- Below the table, a small "Coincidences" summary: count of events in
  the active stream that have a match in any pinned stream within
  `coincidenceWindow`.

### Timeseries event markers

- Existing event markers (per channel on timeseries) extended to render
  ALL pinned streams, color-coded per stream.
- Coincidence detection runs client-side on the event window
  (`useEventWindowQuery` already pulls a window around the cursor; do
  the matching there).
- A `coincidence` glyph (e.g. ringed marker, or thick vertical bar
  joining the two streams' markers) at any pair of events within
  `coincidenceWindow` seconds.

### Server

- Existing `/api/v1/events/{name}/window` endpoint stays as-is. The
  frontend just fetches multiple streams in parallel via React Query.
- No new server endpoints in v0.

## Tests

- Frontend: coincidence detection logic — synthetic 2-stream payloads,
  assert correct pairing within / outside the window.
- Light visual test: store mutations, pinned-stream toggle.

## Acceptance

1. With two streams pinned, both render markers on the timeseries.
2. Switching streams in the event table doesn't reset the timeseries
   marker overlay.
3. Coincidence count is accurate (asserted in the test).
4. Per-stream color comes from the existing colormaps registry, not
   hardcoded.

## Out of scope

- Server-side coincidence detection (large-N or multi-tensor case).
  Defer until a 100k-event dataset shows the client-side path is too
  slow.
- Cross-channel coincidence (this stream's event on ch5 vs that
  stream's event on ch7 considered a match). v0 only matches by time.
- 3+ stream coincidence (e.g. "fires in all 3 within 100 ms"). v0 only
  does pairwise.
