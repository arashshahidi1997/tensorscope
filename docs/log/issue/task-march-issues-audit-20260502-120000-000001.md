## Findings

### Note 1 — `issue-arash-20260312-155321-891713.md`

- [landed] Spectrogram time-axis labels — `SpectrogramView.tsx:188-189` renders `XTicks` + "Time (s)" axis label.
- [landed] Spectrogram time-axis gestures (drag-pan, scroll-zoom, box-zoom) — `useHeatmapGestures.ts` provides pan/zoom/wheel modes; toolbar in `SpectrogramView.tsx:127-148` exposes them.
- [landed] Frequency-axis gestures (Y axis on heatmaps) — `useHeatmapGestures.ts:193-205` handles vertical-only zoom; wheel zoom on Y at `:226-230`.
- [landed] Heatmap-on-plot directional gestures (horizontal drag → zoom X, vertical → zoom Y, both → box zoom) — `useHeatmapGestures.ts:178-207` has explicit `isHorizontal`/`isVertical`/box branches.
- [landed] Spatial maps NOT zoomable — `SpatialMapSliceView.tsx` and `PSDSpatialView.tsx` use only click + hover (no `useHeatmapGestures`).
- [partial] Transform parameter "0 default, can't backspace" — fixed in DAG transform picker (`DAGGraphView.tsx:614-637` uses `type="text"` + `inputMode="decimal"` with raw-string handling and `null` for empty). Still broken in `ProcessingPanel.tsx` for fields with hard defaults: notch_q (`:221`), bandpass_order (`:156`), notch_harmonics (`:198`), spatial_median_size (`:246`) all use `type="number"` + `parseFloat(...) || default`, which snaps back to default on partial input.
- [landed] Side-by-side / top-bottom panel layout for signal + transformed tensors — `viewGridLayout.ts` defines 3 stacked rows (signal, psd, spectrogram), each with side-by-side slots; `ResizeHandle` allows resizing rows and slots.

### Note 2 — `issue-arash-20260313-141053-900146.md`

- [invalidated] PSD spatial map shows 8x8 instead of 16x16 on demo2 — current bundled demo (`scripts/generate_demo_data.py:39-40`) uses `nap=8, nml=8`; "demo2" 16x16 fixture is no longer in the repo. `extractPSDSpatialAtFreq` (`frontend/src/api/arrow.ts:379-428`) infers grid size from received cells, so the renderer faithfully shows whatever the server returns. Cannot reproduce without restoring a 16x16 dataset.
- [partial] AP/ML axis labels consistent across spatial maps — both views render the same `AP` (Y) and `ML` (X) directional labels (`SpatialMapSliceView.tsx:158,182`; `PSDSpatialView.tsx:118,128`), but both leave the numeric tick containers empty (`<div className="axis-y-ticks" />` / `axis-x-ticks`), so 0–N row/col indices are not shown on either view.
- [landed] Y-axis shift+scroll for gain (no button) — `TimeseriesSliceView.tsx:159-168` — Shift+wheel anywhere on the chart adjusts `gainMultiplierRef`.
- [partial] Y-axis drag icon mismatched to actual gesture — there is no Y-specific drag icon any more; `ChartToolbar.tsx` exposes A / ± / ⇕ Y-mode buttons (auto/fixed/fit). Y pan via drag is not implemented (the global "pan" tool only shifts X — `TimeseriesSliceView.tsx:95-100`), so there is no longer an icon-vs-gesture mismatch, but Y-drag itself is still missing.
- [open] Click-and-drag pan on axes only (X for time, Y for amplitude), NOT on the plot itself — global "pan" tool acts on the whole `over` element and only affects the X scale; no axis-only hit testing or Y-pan.
- [partial] Time-selection input next to time-scale control — `TimeScaleBar` (`ChartToolbar.tsx:106-170`) renders the `ts-time-input` and supports `onJumpToTime`. But `SpectrogramView.tsx:194` (the only caller in scope) doesn't pass `onJumpToTime`, so the input only displays the cursor — pressing Enter does nothing. `TimeseriesSliceView` doesn't render `TimeScaleBar` at all.
- [landed] Horizontal time slider below the timeseries view — `NavigatorView.tsx` is the thin overview chart with drag-to-zoom and click-to-select; lives in the bottom panel via `LayoutShell` and serves as the navigation slider.
- [partial] Sidebar text overflow ("bandpass filter order") — sidebar is now resizable 240–480 px (`layoutStore.ts:76-83`, default 260); `.param-label` is `display:block` so labels wrap. The "ord" + 40 px box on `param-row` (`ProcessingPanel.tsx:148-157`) fits at the default width. Acute overflow probably resolved by resize handle, but no per-field test confirms it.
- [partial] Decimal-point typing in transform parameter inputs (".95") — fixed in DAG transform picker (`DAGGraphView.tsx:614-637`, raw-text + parseFloat). Still broken in `ProcessingPanel` `type="number"` fields where `parseFloat("") || default` snaps back to the default before the user can type the digits after the dot.
