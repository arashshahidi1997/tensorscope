**Severity:** medium (benign warning, but cascades on every SSE event and burns CPU)
**Status:** **Fixed in commit `0b4d5b3`** (2026-05-08). Pre-existing — dates to M7 (commit 5c2f5c52, 2026-03-12); surfaced by playwright console-capture during the agent-pairing v1.x smoke (2026-05-08).

## Resolution

Fix shipped per the suggested approach with one addition: `onCommitSelection` is read through a ref (it's a fresh arrow on every App render — `App.tsx:68` wraps the mutation without `useCallback`) so it stays out of the memo dependency tuple. `selectionDraft` is now memoised on the five underlying primitives (`timeCursor`, `freq.freq`, `spatial.{ap,ml,channel}`), and the navigator JSX itself is memoised with explicit deps. The effect now depends only on the memoised JSX's identity, so React's `useState` `Object.is` bail-out kicks in and the parent stops re-rendering on no-op pushes.

Frontend tests: 126 passing. Backend pairing tests: 14 passing. Awaiting live console-error verification from pixecog playwright.

## What

`WorkspaceMain.tsx:231–267` runs a `useEffect` with **no dependency array** that calls `navigatorRef.current(<JSX>)` on every render. The render prop sets state in the parent (`LayoutShell` bottom-panel slot), which triggers a re-render of `WorkspaceMain`, which re-fires the effect, which sets state again — until React's safety threshold caps the loop and emits:

```
Warning: Maximum update depth exceeded. This can happen when a component
calls setState inside useEffect, but useEffect either doesn't have a
dependency array, or one of the dependencies changes on every render.
    at WorkspaceMain (.../components/views/WorkspaceMain.tsx:88:33)
    at ErrorBoundary
    ...
    at QueryClientProvider
```

Captured 14× during the pairing v1 smoke (every ~30 s, on each SSE-driven `selection_changed` / `viewport_changed`). Visible UX is fine — React's cap absorbs the loop — but every external state push triggers a thrash that recomputes / re-renders the navigator subtree thousands of times.

## Where

`frontend/src/components/views/WorkspaceMain.tsx:231–267`

```tsx
useEffect(() => {
  if (!navigatorRef.current) return;

```

Console log: `/storage2/arash/projects/pixecog/.playwright-mcp/console-2026-05-08T14-56-17-267Z.log`

## Why this is fresh now

The bug existed since M7 but was rarely user-visible because human navigation produces few enough re-renders to stay under React's safety threshold per render. SSE-driven mutations from a paired agent push state mutations from outside the React commit phase, which cascade into more re-renders, which now reliably exceeds the threshold.

## Suggested fix (not done in this commit)

Lift the JSX into a `useMemo` with proper deps, then call `renderNavigator` from a separate effect that depends only on the memoised element:

```tsx
const navigatorElement = useMemo(() => {
  if (!navigatorData) return null;
  return (
    <>
      <NavigatorView slice={navigatorData} ... />
      <TimeScaleBar ... />
      {showHypnogram && brainstateAvailable && brainstateIntervals.length > 0 && (
        <HypnogramView ... />
      )}
    </>
  );
}, [
  navigatorData, selectionDraft, setTimeWindow,
  brainstateIntervals, brainstateOverlay, brainstateAvailable,
  viewportDuration, setViewportDuration, showHypnogram,
  brainstateTimeRange[0], brainstateTimeRange[1],
  timeWindow, onCommitSelection,
]);

useEffect(() => {
  navigatorRef.current?.(navigatorElement);
}, [navigatorElement]);
```

Caveat: the inline-arrow callbacks (`onSelectTime={(t) => onCommitSelection(...)}`) close over `selectionDraft`, which is a fresh object every render (it's `toSelectionDTO(selectionState)`). So even with `useMemo`, the dep tuple changes on most renders — won't loop, but won't skip either. Two cleaner paths:
- Memoise `selectionDraft` itself with `useMemo` keyed on the underlying primitive store fields.
- Refactor `onCommitSelection` so it reads the latest selection from the store inside, removing the need for `selectionDraft` to be a callback dep.

## Out of scope for this issue

- The `setSelectedTensor` `useEffect` at line 114 looks fine (deps `[selectedTensor, stateQuery.data?.active_tensor, setSelectedTensor]`).
- The `setWorkspaceObjects` effect at line 120 has deps but recomputes the entire array each render — not infinite-loopy because `setWorkspaceObjects` does referential-equality bail-out internally? Actually no, a plain Zustand setter isn't bail-safe; this effect re-runs whenever `stateQuery.data?.tensors` reference changes, which is per refetch, which is acceptable.
