Pairing v1 follow-up: viewport state vs selection state — separate concerns, both need API surface

**Context.** During pixecog's joint-test smoke (2026-05-08), `set_selection(time=120.0)` updated the EventTable time indicator (read selection.time) but did NOT pan the timeseries viewport, which stayed in the first ~10 seconds. SSE broadcast `selection_changed` correctly — the gap is that viewport-bound views (timeseries / spectrogram x-axis / navigator) don't react to selection.time changes coming from the API the way they react to a user EventTable click.

**Underlying distinction** (from arash, 2026-05-08): long axes have two state items that should be modelled as separately controllable:

| State | Concept | UI example |
|---|---|---|
| **Viewport window** | What time range is visible (`t_lo`, `t_hi`) | timeseries x-axis bounds, navigator strip extent |
| **Selected time** | Where the cursor / playhead is drawn | red vertical, EventTable time indicator, freq-cursor |

These are deliberately independent in many tools — keeping a fixed viewport while moving the cursor is a useful pattern (e.g., reading off "what is at t=120?" without losing the surrounding context you've already framed). Linking them is also useful (jump-to-cursor when out of viewport — Ableton, video players, neuroscope2 default).

**Concrete API gaps in v1.**

1. No `set_viewport(t_lo, t_hi, ...)` primitive. An agent that wants to scroll the timeseries to a specific time range can't.
2. No `follow=True` flag on `set_selection`. Convenience for "pan window to keep cursor visible" — the most common use.
3. Convention for which views are viewport-bound vs cursor-bound isn't documented. Right now EventTable consumes selection.time directly; timeseries seems to consume viewport.t_range only and ignores selection.time changes (no auto-pan, even on EventTable click — TBD by user).

**Proposed addition for v1.x.**

```python
# selection cursor only — current behaviour
ctx.set_selection(time=120.0)

# viewport pan — new
ctx.set_viewport(t_lo=115.0, t_hi=125.0)        # explicit
ctx.set_viewport(t_center=120.0, t_window=10.0) # convenience

# combined — most common interactive pattern
ctx.set_selection(time=120.0, follow_with_window_s=10.0)
```

Mirror for spatial: `set_viewport(ap_range=..., ml_range=...)` if/when spatial views support pan/zoom.

**Why this matters for the spindle-validation use case** (pixecog idea-arash-20260507-161642-673518):
- The agent computes a fresh spectrogram for one ~20 s NREM bout, injects it as a tensor, and wants the human to see exactly that 20 s span — not jump-to-cursor at one timepoint inside it. `set_viewport(t_lo, t_hi)` is the natural verb.
- The agent suggests "the highest-prominence event is at t=215.4 s" — `set_selection(time=215.4, follow_with_window_s=4.0)` is the natural verb (cursor + auto-pan to ±2 s context).

**Frontend plumbing implications.**
- React store needs separate slices: `selectionStore` (cursor) and `viewportStore` (window bounds), with each view subscribing to whichever it needs.
- SSE channel needs a new event type: `viewport_changed` alongside `selection_changed`.
- Per-view viewport state (timeseries.t_range, spectrogram.t_range) may already exist client-side; the question is whether to lift it to the Python core (so agents can drive it) or expose a passthrough API.
