**Severity:** medium — blocks long-window panning of the new `spectrogram_live` view, but works fine at workflow-relevant 1–10 s windows.
**Status:** open. Filed during the Phase A live-test wrap-up (2026-05-08); spec_live itself is functional end-to-end at small windows.

## What

`spectrogram_live` on a `(time, AP, ML)` LFP tensor returns a `(time, freq, AP, ML)` slice. With default params (`nperseg_s=1.0`, `noverlap_pct=95`, `fmin/fmax = 0.5..30`), the cell count grows roughly as:

```
cells = n_time_segments × n_freq × n_ap × n_ml
      ≈ (window_s / 0.05) × 30 × n_ap × n_ml      # 0.05 s segment hop at 95% overlap
```

For the standard sub-01/ses-04 16×16 LFP grid:

| Window | n_time_segments | Cell count | Render |
|---|---:|---:|---|
| 8 s  | ~156 | ~1.2 M | renders cleanly |
| 40 s | ~780 | **~6 M** | browser tab crashes |

The crash arrives **after** the server has returned 200 OK with the full payload — i.e., the wire is fine; the choke is rendering each AP×ML tile through SpectrogramView's canvas path independently and React/uPlot can't keep up.

## Repro

1. Paired session against the standard sub-01/ses-04 LFP tensor (`pixecog/.../tensorscope_launch.py --pair`).
2. From a Python REPL:
   ```python
   from tensorscope.pairing import get_context
   ctx = get_context()
   ctx.set_viewport(t_lo=200.0, t_hi=240.0)
   ```
3. Observe the browser tab crash within ~5 s of the slice response landing.

A `8 s` window (e.g., `set_viewport(t_lo=211, t_hi=219)`) renders cleanly — see `pixecog/.playwright-mcp/spec-8s.png`.

## Why now

`spectrogram_live` is the first view in the codebase that returns a *4-D* slice (time × freq × AP × ML) for direct on-screen display. The pre-existing `spectrogram` view assumes the server already collapsed to 2-D. The frontend's `extractSpectrogram` averages over trailing spatial dims, but the canvas / React path between Arrow decode and pixel commit still scales with the full cell count.

## Suggested fixes (any one, in increasing order of plumbing depth)

1. **Time-bin downsampling on the server.** Cap `n_time_segments ≤ min(window_s × 25, native_n_segments)` inside the `spectrogram_live` branch in `apply_slice_request`. Report the effective time resolution via the existing `spectrogram_live_*` attrs. Cheapest fix; keeps the full freq/spatial detail. Numbers: at 25 segments/s, a 40 s window → 1000 segments → ~480 k cells. Fine.

2. **Spatial aggregation in the response.** When `n_ap × n_ml × n_time × n_freq > ~1 M`, average over an AP/ML stride so the returned grid is at most ~8×8. Costs spatial detail; useful when zoomed out.

3. **Per-tile virtualisation in SpectrogramView** when `grid_size > 4×4`: only render the AP×ML panels currently visible. Bigger refactor; pays off for any view that returns spatial fan-outs.

4. **Frontend escape hatch:** detect the tile-grid size at decode time and switch to a "mean over spatial dims" render mode (single 2-D heatmap, like the precomputed `spectrogram` path). Loses spatial detail but renders instantly. Could be a temporary stopgap before either (1) or (2) lands.

(1) is the trivial pick if we want a same-day fix. (3) is the right long-term answer for any 4-D-payload view.

## Out of scope here

- Phase B parameterised-live-view API (`add_live_view` / `update_live_view`) — orthogonal.
- The `spectrogram_live` compute path itself: server returns the data correctly and quickly. The choke is purely client-side.
