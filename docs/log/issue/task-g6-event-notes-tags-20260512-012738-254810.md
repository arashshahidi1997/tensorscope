# G6 — per-event annotation layer

## Why

From the spindle-review critique
(`docs/log/issue/issue-arash-20260511-182119-502518.md` G6): "This one
is a beautiful example for the methods figure" — there's no way to mark
that. The event-review store already has `notes` and `tags` slots in
its schema but no UI to edit them.

## What to build

UI for editing per-event notes and tags. The store already supports
both fields (see `frontend/src/store/eventReviewStore.ts` —
`updateNotes` and the `tags?: string[]` field on `EventDecision`).

### EventTableView

- Click the status badge in the existing event table to open an
  inline popover with:
  - Status radio (Accepted / Rejected / Maybe / Pending)
  - Notes textarea (free-form text)
  - Tags input (comma-separated chips)
- Popover commits on blur / Escape / Cmd+Enter.
- Notes column in the table shows the first ~30 chars of the note
  with a tooltip for the full text.

### Store extensions

- New action `updateTags(key, tags: string[])` — minimal addition.
- `updateNotes(key, notes)` already exists.
- Persist (already wired via Zustand `persist` middleware).

### Keyboard

- `e` opens the popover for the current event (and focuses the notes
  textarea). Matches Neuroscope2's "edit cluster" gesture.
- Escape closes without commit if no changes.

### Tests

- Store: `updateTags` round-trips correctly, preserves notes/status.
- Popover: smoke test that opening/closing doesn't fire stray decisions.

## Acceptance

1. Click badge → popover opens with current status/notes/tags.
2. Edit notes → blur → table row shows truncated note preview.
3. Reload → notes/tags persist.
4. The keyboard `e` shortcut bails inside inputs (same guard as the
   existing event-review shortcuts).

## Out of scope

- Server-side note storage — that's G9 (validation export).
- Markdown rendering in notes — plain text only in v0.
- Tag autocomplete from prior-used tags — defer.
