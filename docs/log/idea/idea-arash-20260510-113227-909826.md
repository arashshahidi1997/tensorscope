# Non-grid probe geometries + 3D atlas registration

Two related threads the user flagged on 2026-05-10 as needing
**study + planning + spec writing**, not implementation.

## Thread 1 — non-grid channel layouts

**Already covered.** [`docs/design/probe-layout.md`](../../design/probe-layout.md)
is a committed proposal (multi-week effort) for a `ProbeLayout` sidecar
on `ServerState`, with canonical `(time, channel)` and the current
`(time, AP, ML)` grid demoted to a renderable special case. It
addresses tetrodes, Neuropixels (`time, depth` or `time, shank, depth`),
laminar electrodes, and heterogeneous multi-region recordings.

User's prompt phrased this as "neuropixels (1-D approximation of the
checkerboard channel arrangement) or generally other channel layouts".
That's exactly what `ProbeLayout` is designed to absorb.

**Nothing new to write here.** Open question is just *when* to schedule
the implementation. The probe-layout doc is decision-ready.

## Thread 2 — 3D rat brain atlas (NEW)

User wants programmatic access to a 3D rat atlas, with Long-Evans rats
in mind (the lab's strain). The ChatGPT research dump in the prompt
surveys the options:

### Atlas options surveyed

| Source | Access | Best for |
|---|---|---|
| **BrainGlobe Atlas API** (`whs_sd_rat_39um`) | `pip install brainglobe-atlasapi` | local Python, region masks, meshes, annotation volumes |
| **EBRAINS / siibra** | `pip install siibra` + REST API | atlas-linked multimodal data, HTTP service |
| **Native WHS NIfTI** | nibabel + DataLad | reproducible pipelines, ITK-SNAP / ANTs / FSL |
| **Scalable Brain Atlas** | web service | reference / coordinate transforms |

The canonical reference is **WHS — Waxholm Space Sprague-Dawley rat
atlas, 39 μm, 222 annotated structures**, available via BrainGlobe as
`whs_sd_rat_39um`.

### Long-Evans applicability — the caveat

WHS is built from an 80-day-old **male Sprague-Dawley** rat. For
Long-Evans subjects it's acceptable as a *reference* atlas (region
identity, electrode-to-region mapping, visualization, mask derivation),
but **coordinates are not strain-exact**. Best-practice flow:

```
Long-Evans subject MRI / histology / CT / electrode reconstruction
  → register to WHS Sprague-Dawley atlas
  → use WHS labels through the transform
  → report registration QC
```

Long-Evans-specific MRI templates exist on NITRC (4 male LE rats, 8wk,
T2-weighted, intended for SPM normalization) but are not as broadly
used for detailed labeling. Practical recommendation: use WHS as the
**common annotation space** and phrase results as "registered to WHS
SD rat atlas".

## Where this intersects ProbeLayout

Atlas integration is a *natural extension* of `ProbeLayout` rather than
a separate system:

1. **Per-channel 3D positions** — ProbeLayout already carries
   electrode geometry "in 2D or 3D" (per the design doc §1). 3D
   positions are the entry condition for atlas registration.
2. **Per-channel anatomy** — ProbeLayout already carries channel
   groups with "region, anatomy, colour". Today these are
   user-supplied strings. With atlas registration they could be
   *derived* from the WHS annotation volume by sampling
   `annotation[round(zyx)]` per channel.
3. **Reference scheme** — once channels are tagged with
   atlas-derived regions, the per-group CMR scheme (already in the
   design doc) can be auto-populated by region instead of by manual
   shank assignment.
4. **View extensions** — a new view type ("anatomy_3d" or
   "atlas_slice") could show channels-on-atlas, leveraging
   `brainrender` or a custom Three.js renderer in the frontend.
5. **Selection by region** — `SelectionDTO` could grow a
   `region: str | None` field that filters channels by their
   atlas-derived region label.

## Proposed deliverables (when the user says go)

A markdown design proposal at `docs/design/atlas-integration.md`
covering:

1. **Scope decision** — read-only annotation lookup, vs. full
   registration pipeline (subject MRI → WHS), vs. just bundling a
   pre-registered atlas as reference geometry. Pick one. The first
   is days of work; the third is multi-month.
2. **Backend dependencies** — `brainglobe-atlasapi` is small and
   pure-Python-ish; `siibra` pulls a heavier stack. Decide whether
   to bundle (default) or make optional.
3. **Atlas data lifecycle** — BrainGlobe caches to
   `~/.brainglobe/`; for reproducibility we likely want a project-
   level cache under `data/atlas/` with explicit version pinning.
4. **ProbeLayout schema delta** — what fields does ProbeLayout gain?
   `coordinate_space: str` (e.g. `"WHS_SD_rat_39um"`), per-channel
   `region_id: int | None` and `region_name: str | None`,
   `transform_to_atlas: 4x4 affine | None`.
5. **API surface** — new endpoints? `GET /api/v1/atlas/regions`,
   `GET /api/v1/atlas/regions/{id}/mesh`,
   `POST /api/v1/probe/{id}/register-to-atlas`.
6. **Long-Evans handling** — explicit policy: "WHS SD is the
   reference; LE subjects require registration; we never report raw
   LE coordinates as WHS coordinates". Document the QC report.
7. **Out of scope** — full registration algorithms (use ANTs/FSL
   externally), other species, behavioural tracking volumes.
8. **Effort estimate** — surgical / multi-week / multi-month per
   chosen scope.

## Open questions for the user

- **Existing data?** Does the lab already have WHS-registered
  electrode coordinates for any Long-Evans subjects, or would we be
  starting from raw histology / CT?
- **Sequencing.** Atlas integration depends on ProbeLayout's
  `(time, channel)` pivot landing first. Should we (a) do
  ProbeLayout, then atlas; (b) bundle them so the ProbeLayout
  schema is designed with atlas in mind from day one; or (c) keep
  atlas as a true sidecar that any future ProbeLayout can opt into?
- **Read-only vs registration pipeline.** Does TensorScope need to
  *do* the registration, or can we assume registered coordinates
  arrive via the data loader?
- **Visual priority.** Is the killer feature (i) anatomical labels
  on existing 2-D views, (ii) a new 3-D atlas-aware view, or (iii)
  region-based filtering and grouping?

## Decision

No code. This is captured as a planning artifact pending user
direction on scope and sequencing. The ProbeLayout design doc is the
gate — atlas integration should not start until ProbeLayout's
implementation is at least underway, otherwise the schema decisions
collide.
