Idea: tensorscope agent-pairing API à la marimo `_code_mode`

**Context (from pixecog 2026-05-07 session).** While auditing detected spindle events on sub-01/ses-04 ECoG, we tried to use `tensorscope_launch.py` (the pixecog CLI) to view event blobs overlaid on the spectrogram. Two practical blockers surfaced:

1. The launcher reads from `manifest_assemble/events/...parquet` and `spectrogram_burst/spectrogram/spectrogram/...spec.zarr` — both can be stale or missing, requiring a pipeline re-run before tensorscope is useful.
2. Once tensorscope is up, we (Arash + claude pair) can navigate channels/time interactively but **cannot programmatically tweak** spectrogram parameters, detector thresholds, or normalization modes from within tensorscope. To iterate, we ended up writing a marimo notebook that exposes UI sliders for MT-window, overlap, bandwidth, h-threshold, min_area, etc., and renders the spec/raw/bandpass + event overlay live.

**Idea.** Add a tensorscope-side equivalent of marimo's `_code_mode` — a programmatic API that lets a paired agent:
- read the current frame state (selected channel, time window, active tensor, active event stream)
- mutate selection/zoom/pan
- inject computed tensors (e.g., a freshly computed spectrogram with new MT params) into the live view
- inject event streams (e.g., events from a re-tuned detector) without re-bundling
- read screenshot/tensor data back for the agent to validate

**Concrete UX target.** From a paired session, the agent can do: "compute a new spectrogram on this channel with bandwidth=4 Hz, run h-maxima at h=3.0, replace the active spectrogram + event overlay, take a screenshot, evaluate whether the events look spindle-like." Today this requires `pipeio_run` on the cohort + `manifest_assemble` re-run + tensorscope re-launch — a 30-min loop per parameter. With the proposed API, it'd be 2-3 seconds.

**Why tensorscope is the right home.** Marimo gives us live re-execution but the visualisation surface is matplotlib — fine for snapshots, weak for interactive multi-panel browsing. Tensorscope already has SpectrogramView, HypnogramView, EventTableView, SpatialEventView, OrthoSlicerView, NavigatorView, PSDCurveView/HeatmapView. If those views could be programmatically driven by an agent (not just by the human via UI), the paired-detection-tuning workflow becomes natural inside tensorscope.

**Possible scope.**
- v1: server-side WebSocket or HTTP RPC channel exposing read/write of view state (selection, active tensors, active event streams).
- v2: in-process Python API alongside `tensorscope serve` (similar to how marimo has `_code_mode` accessible from the kernel).
- v3: agent-pairing skill (à la `marimo-pair`) that auto-discovers a running tensorscope and exposes a `with cm.get_context() as ctx: ctx.set_selection(...), ctx.add_tensor(...), ctx.add_events(...)`.

**Cross-ref.** pixecog session 2026-05-07 — see [docs/log/result/result-arash-20260507-* spindle audit notebook] when filed; the marimo-driven workflow at `pixecog:code/pipelines/spectrogram_burst/notebooks/explore/spindle_detector_tuning.py` is the working baseline an agent-pairing-tensorscope would replace.

## Design discussion: is this a tensorscope flaw, or an extension?

(Added 2026-05-07 as follow-up to the same session.)

**Not a flaw — a tradeoff that didn't have agent pairing as a first-class
constraint at migration time.** The Python-core / React-frontend split was
the right call for human UX (Panel/HoloViews can't deliver the multi-pane
interactive feel of tensorscope's current frontend, and rendering 100M-cell
tensors in matplotlib is hopeless). The cost: a Python ↔ JS divide that
agents can't cross trivially.

**Where tensorscope is already right for this:**

- `tensorscope.core` (state / schema / layout / data / events) is *already*
  a structured, data-first model. That's the hard part of the redesign,
  and it's already done.
- Selection, tensor registry, event registry are Pydantic-typed in Python —
  they don't have to be reverse-engineered from React store internals.

**Where the gap is:**

- View state propagation is one-way: human acts in React → backend gets
  notified → backend updates Python state. There's no equally-loud reverse
  channel: backend mutation → frontend re-render.
- Per-view ephemeral state (zoom level, hover, cursor) lives only in
  React. Fine for the human, opaque to agents.
- "Render-back" capability (screenshot to agent) doesn't exist. Marimo
  dodges this trivially — matplotlib output is a Python object that
  returns straight to the kernel.

### How I'd extend it — closest to marimo's mental model

| Layer | Role | Source of truth |
|---|---|---|
| **Python core** | Tensor registry, event registry, layout, **authoritative selection state** (active tensor, current time window, active view, color scale, freq band) | Yes |
| **WebSocket bus** | Bidirectional sync of authoritative state. State mutations on either side get broadcast. Agents and React frontends are peers — both subscribe, both can mutate. | — |
| **React frontend** | Renders authoritative state + holds **ephemeral UI** (mouse position, hover, transient zoom rubber-band). Sends authoritative mutations through the bus. | Only for ephemeral state |
| **Agent code-mode** | `tensorscope._code_mode.get_context()` — same shape as marimo's: read state, mutate state (via the bus), inject tensors / events into the registry, request screenshot. | — |

The key principle: **the Python core becomes a shared session, not just a
data server.** Multiple clients (one human in browser, one agent scripting)
subscribe to the same state stream. Marimo notebooks proved this works; the
difference is that marimo bundles the renderer (Jupyter-style) and
tensorscope already split it (good for performance).

### Concrete additions over current arch

1. **WebSocket subscription endpoint** that streams authoritative-state
   diffs (selection changes, tensor registry changes, event registry
   changes). Both React and Python clients (agents) subscribe.
2. **State-mutation API** at the Python core (already mostly there — just
   needs to broadcast on mutation).
3. **`tensorscope._code_mode` Python module** modeled on
   `marimo._code_mode`: `get_context() → ctx.list_tensors(),
   ctx.add_tensor(name, da), ctx.add_events(...), ctx.set_selection(...),
   ctx.read_selection(), ctx.snapshot()`.
4. **Server-side render hook per View** for screenshots (e.g., a
   Plotly/Vega snapshot endpoint, or a headless-Chrome puppeteer wrapper
   for React-rendered views — the latter is the most practical
   near-term).

### What I'd NOT do

- Re-render in Python (would lose tensorscope's UX gains).
- Replicate state via CRDT (overkill for single-author session pairing —
  last-writer-wins is fine).
- Run tensorscope in Pyodide (deployability/perf nightmare, and won't
  work for the GB-scale tensors that motivated the split in the first
  place).

### Honest summary

The architecture is **80 % there for agent pairing**. The missing 20 % is
plumbing — a state-sync bus and a Python-side `_code_mode` analog. Not a
redesign, an extension.

**Difficulty per capability:**

| Capability | Difficulty | Why |
|---|---|---|
| Inject computed tensor (agent computes a fresh spectrogram, hands it to tensorscope to display) | **Easy** | Backend has the registry; an HTTP `POST /tensors` endpoint plus the existing TensorChooser already picks it up. |
| Inject event stream (events from a re-tuned detector) | **Easy** | Same — already a registry; EventTableView / SpatialEventView consume it. |
| Read current view state (what is the human looking at?) | **Easy-medium** | If selection state lives in the Python core (it does), HTTP GET works. If only in React, need WebSocket subscription. |
| Mutate selection from agent → frontend updates | **Medium** | Needs backend → frontend push channel (WebSocket / SSE) wired into the React store. |
| Drive view-specific params (e.g., adjust spectrogram colormap, freq axis range) | **Medium-hard** | Depends on whether the param mutates a Python-side render config or a React-only display setting. If client-only, agent can't drive without frontend cooperation. |
| Screenshot back to agent for visual validation | **Hard** | Needs server-side rendering OR a headless browser wrapper. Marimo dodges this by being a Python kernel. |

**For the spindle-detection workflow specifically, v1 alone would already
replace the marimo notebook** for cases where tensorscope's existing views
beat matplotlib (multi-channel browsing, hypnogram, spatial map). The
agent does the compute, hands tensors+events over, the human navigates.
