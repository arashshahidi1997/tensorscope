Use case: human + agent interactive validation of detected sleep events (spindles, slow waves, ripples)

This is a **concrete workflow spec** distilled from the pixecog 2026-05-07
session, where we tried (and failed) to use tensorscope to validate
spindle blob events. Sister to the agent-pairing API idea (note
20260507-160104-478773); this one is the use-case-driven feature list,
not the architecture.

## Who / what / why

**User:** an electrophysiology PhD/postdoc + a coding agent in a
paired session, validating events emitted by an automated detector
before trusting downstream analyses (CCGs, peri-event histograms,
coupling figures).

**Why this matters:** detector parameters are usually tuned blind from
papers — but cohort-specific noise floors, recording quality, and
brain-state coverage mean a detector that looked fine in the literature
can produce thousands of false positives or miss real events on a
given dataset. A trustworthy CCG starts with detector validation, and
detector validation starts with eyeballing per-event waveforms +
spectrograms in context.

The pixecog detector ran today: 35,334 spindle events on sub-01/ses-04,
of which 92.5% land in NREM (good biological specificity) but with
prominence distribution heavily concentrated at the noise floor —
suggesting the detector is firing on real spindles AND on band-noise
fluctuations indistinguishable from them. Without an interactive
validation tool, we cannot tell which events to keep.

## The current workflow gap

What we ended up building (in marimo) because tensorscope wasn't
ready:
1. Slider UI for spectrogram + detector params (MT window/overlap/
   bandwidth, band, h, min_area, min_duration, normalization,
   NREM-restrict).
2. Per-channel multitaper spectrogram renderer (turbo cmap, Prerau-
   style per-frequency baseline subtraction).
3. Event overlay on the spectrogram with bbox markers.
4. Bandpass + Hilbert envelope panel for the same channel/window.
5. 16×16 spatial event-count map for the current window.
6. Hypnogram strip (Wake/REM/NREM colored bands).

That works for one channel at a time, with matplotlib re-renders. It
breaks down for: cohort-wide multi-channel browsing, smooth scroll-
based time navigation, programmatic agent control of
visualizations, side-by-side detector-variant comparison, and human
event-by-event annotation.

Notebook reference:
`pixecog:code/pipelines/spectrogram_burst/notebooks/explore/spindle_detector_tuning.py`

## Ideal session — workflow narrative

T+0: User opens tensorscope. The session auto-discovers all
spectrogram_burst detection outputs for the active subject/session and
registers them as available event streams (legacy-z3, h-maxima blobs,
mtm variant, etc.).

T+0:30: **Spatial view** opens. 16×16 ECoG grid colored by event-count
per channel for the current time window. User sees a U-shape (corner
channels firing 3× the interior median). Agent annotates the chat:
"that's an h-maxima edge artifact; want me to re-run the detector
with edge channels excluded?". User clicks "interior only" toggle.

T+1:00: User scrolls the timeline. **Spectrogram view** updates
smoothly (uplot-style interactive zoom/pan); event markers move along.
Hypnogram strip below reminds the user where NREM bouts are.

T+2:00: User clicks an event in the table. Tensorscope jumps the time
cursor to the event peak and pulls up a **per-event audit panel**: ±2 s
windows of (raw broadband, 10–16 Hz bandpass + Hilbert envelope,
multitaper spectrogram with blob bbox overlay), all on aligned axes.
User marks the event as "looks like a spindle" via a one-key shortcut.

T+3:00: Agent suggests a tuning experiment: "let me re-run h-maxima
with h=3.0 and min_duration=0.5 s; should be ~20% as many events but
cleaner." Agent runs the detector in the Python kernel (tensorscope-
adjacent), then **injects the new event stream as a parallel layer**.
Both event streams are visible on the spectrogram with different colors
(orange = h=1.0 legacy, cyan = h=3.0 candidate).

T+4:00: User flags 30 events across both streams as
"spindle/not-spindle" via the event-table UI. Agent reads the labels,
computes precision per detector variant on the labelled subset, posts
the score: "h=1.0 → 0.34 precision (your manual labels). h=3.0 → 0.81
precision, but you missed 6 weak events that did look like spindles
visually. Recommendation: h=2.0 with min_duration=0.6 s."

T+5:00: Agent re-runs at the recommended params, replaces the
candidate stream. User scrolls a few NREM bouts, flags 10 more events,
agent recomputes. After 3 iterations user is satisfied.

T+10:00: User saves the **tuning checkpoint** — a small JSON capturing
detector params, normalization mode, manual labels, and selection
state. Filed under `derivatives/spectrogram_burst/checkpoints/` as
provenance for downstream CCG analyses. Agent attaches a result note
in pixecog notio with the checkpoint reference.

## Required tensorscope capabilities (priority-ranked)

Priority labels: **P0** (blocks the workflow), **P1** (would-be-painful
without), **P2** (nice).

### Detection-validation views

- **P0 EventTableView with click-to-jump.** Sortable / filterable by
  event property (prominence, duration, freq_peak, channel, event_id).
  Click row → time cursor + channel selector update across all linked
  views.
- **P0 SpatialEventView (per-channel event count heatmap).** AP×ML
  grid (or any ndim spatial layout) with brightness = events-in-window
  per channel. Click a channel → channel selector update. Lets users
  see where the detector fires *spatially*, which is the single fastest
  way to spot edge / hot-channel artifacts.
- **P0 HypnogramView strip below the time axis.** Wake/REM/NREM as
  colored bands. Optional: brushing the hypnogram restricts the time
  range. Without this, NREM-relevant detectors can't be evaluated
  honestly.
- **P0 Per-event audit panel.** Click an event → pop a side panel with
  raw + bandpass + spectrogram all on the same ±N s window (N
  configurable; default 2 s). The single most important visual aid for
  "is this a real event?" decisions.
- **P1 Multi-stream event overlay.** Two or three event streams on the
  same spectrogram, color-coded, with toggleable visibility per
  stream. Lets users compare detector variants side-by-side.
- **P1 Manual annotation UI.** One-key flag-and-advance: "looks like a
  spindle" / "doesn't" / "edge case", with optional free-text. Persists
  to a sidecar table (sub-XX_ses-YY_detector-DD_labels.parquet).
- **P2 Cohort browser.** A subject-session navigator that loads
  detector outputs for all keepers in the cohort registry and lets
  users hop between them. (Today: requires re-launching the script.)

### Multitaper / parameter exploration

- **P0 Programmatic tensor injection.** Agent computes a spectrogram
  with new MT params and injects it as a named tensor; existing views
  pick it up. No bundle rebuild.
- **P0 Programmatic event-stream injection.** Same for events. After
  agent re-runs the detector, the new stream becomes available as a
  toggleable overlay.
- **P1 Param sliders bound to agent recompute.** A slider in the UI
  changes (say) bandwidth_hz; tensorscope notifies the agent; the
  agent recomputes the spectrogram and re-injects. The user
  experiences live recompute even though the heavy compute is in
  Python.
- **P2 Side-by-side multi-spectrogram view.** Same channel, two
  spectrograms with different params (e.g., scipy vs multitaper, or
  bandwidth=2 vs 4 Hz), aligned time axis, for comparison.

### Provenance and reproducibility

- **P1 Checkpoint export.** "Save current detector tuning state as a
  JSON" — includes detector params, normalization, NREM filter,
  manually-labelled event-ids. Re-loadable as a starting point for
  a follow-up session.
- **P1 Lineage display.** Every active event stream / tensor shows its
  source (file path, code revision that produced it, detector params).
  Helps when an agent injected a tensor 2 hours ago and the user can't
  remember which version it was.
- **P2 Annotation export to notio result note.** One-click "post to
  the project's result-note pipeline".

### Multi-channel browsing (currently the matplotlib bottleneck)

- **P0 Multi-channel band-power heatmap.** Y axis = channel
  (sortable: by AP, ML, by event count, custom); X axis = time;
  colormap = spindle-band power. Event markers as overlay rows. This
  is the view that immediately reveals which channels are quiet, which
  fire constantly, and where in time bursts cluster across the cortex.
  Without this, "are these events real?" requires per-channel
  inspection — minutes per channel, intractable for 256 channels.
- **P1 Channel-cluster grouping.** Group channels by atlas region or
  by user-defined manual clusters, render group-mean band power as a
  separate row.
- **P2 3D propagation view.** Spatio-temporal animation of band power
  across the AP×ML grid for a window — the marimo notebook can't do
  this at all; tensorscope's PropagationView could.

## Anti-requirements (things I am NOT asking for)

- Re-render in Python or Pyodide. The React/Vega/uplot stack is the
  point — keep it.
- Server-side detector execution. Detection is a project-specific
  pipeline; tensorscope shouldn't ship a detector. Just accept event
  streams from outside.
- LLM-in-the-tool. The agent connects from outside via the
  programmatic API; tensorscope is a viewer + state hub, not an LLM
  client. Keeping LLM concerns out of tensorscope is the right
  separation of duties.

## What this would unlock for pixecog

Today: validating 35,334 spindle events on one session means writing
a marimo notebook (1 hour), iterating sliders for half a day, and
accepting that we can't see the whole cohort in one view. Three days
to claim "the detector finds real spindles cohort-wide."

With this set of capabilities: 30 minutes per session of human-time
+ agent recommendation = same claim, with manual labels supporting
it. Across 5 keepers × 4 detector variants, a 2-day budget instead of
a 2-week one.

And once the spindle workflow works, slow-wave validation (lfp_extrema
detector + 2-layer Npx differential vs gamma envelope) and ripple
validation (npx hpc band power) all reuse the same machinery — the
detection-validation workflow generalizes across event types.
