# idea
## M10 candidate: object-centric workspace and propagation UX

### Context
Recent UI work improved performance and waveform fidelity:

- fast Arrow decode in `arrow.ts`
- vectorized backend downsampling in `state.py`
- higher-resolution timeseries and navigator queries
- initial auto-gain support for timeseries
- immediate time-scale buttons for faster zoom feedback

Those changes exposed a larger product issue: TensorScope still behaves too much like a single-signal viewer with add-on panels, while the emerging UX wants an object-centric workspace where source objects, derived objects, processing, propagation, and timeline navigation all feel like first-class concepts.

### Immediate issues surfaced

1. `yAuto` exists conceptually, but its control is not discoverable enough to count as shipped UX.
2. Time-scale buttons feel like zoom deltas instead of deterministic snapped viewport presets.
3. The current "propagation frame" is too narrow and should be replaced by a general propagation mechanism.
4. Processing is still framed too much as "do something to the signal" instead of "apply transforms to any object."
5. The hypnogram strip is not earning its space in the current workflow.
6. The navigator needs a stronger timeline interaction model with a selected-time cursor and drag handle.
7. Multi-object layout is not explicit enough: signal and derived objects should be composable in rows/columns.
8. Channel-stack labeling needs grouped, two-level axis semantics rather than flattened labels.
9. Transform execution is too silent; users need visible progress and completion messages.
10. Apply-in-processing should support `Enter`.

### Product direction
This idea converges on four linked themes:

1. **Object-centric layout**
   Signal, spectrogram, PSD, and future derived tensors should be treated as objects with explicit visibility and layout controls.
2. **View-agnostic processing**
   Processing belongs to any compatible object, not just the loaded signal.
3. **Timeline as a first-class control**
   The navigator should own selected time, snapped view durations, and scrub interactions.
4. **Propagation as a capability**
   Propagation should be a reusable mechanism that can render sampled frames in-place, in a new panel, or as tiled layouts.

### Proposed M10 prompt sequence

| # | Title | Why it exists |
|---|---|---|
| 100 | Object-centric panel chrome and visibility model | Promote `signal`, `spectrogram`, and future derived tensors into explicit workspace objects with object tabs, toggle state, and per-object actions. |
| 101 | Timeline cursor and snapped duration model | Make the navigator authoritative for selected time, draggable time cursor, and exact snapped duration presets. |
| 102 | Timeseries control cleanup | Fix auto-gain discoverability, replace cryptic controls with explicit Y-mode UI, and align timeseries toolbar behavior with the timeline model. |
| 103 | Multi-object layout modes | Add single / row / column object layouts with automatic defaults and thin object labels for docked layouts. |
| 104 | General propagation controller | Replace the special propagation frame with a reusable propagation mechanism for playback, strip, and tiled frame layouts. |
| 105 | Object-scoped processing UX | Attach processing to any object, add keyboard submit on `Enter`, and expose object actions near the object header instead of burying them. |
| 106 | Execution activity and transform feedback | Add visible transform lifecycle reporting for start/progress/finish/error, initially in console and UI activity surfaces. |
| 107 | Channel-stack grouped labeling and track cleanup | Add two-level axis labels for stacked channels and remove low-value surfaces such as the hypnogram strip. |

### Execution order

- **Batch 1**: P101 + P102 + P106
  These tighten the current navigation and transform UX without changing the full workspace model.
- **Batch 2**: P100 + P103
  These establish the object-centric workspace shell and layout semantics.
- **Batch 3**: P105
  This depends on the object model from P100.
- **Batch 4**: P104
  This depends on the object and layout model from P100-P103.
- **Batch 5**: P107
  This cleans up remaining view semantics once the main workspace interaction model is stable.

### Guardrails

- Keep shared selection state global; object visibility and object layout remain view-local workspace state.
- Do not introduce another one-off panel type for propagation.
- Time-scale controls must set exact viewport duration state, not relative zoom deltas.
- Processing must remain object-compatible and transform-driven, not hardcoded to the raw signal.
- Remove or demote low-value UI chrome rather than accumulating optional strips.

### Success criteria

- Users can understand which objects exist and show them in single, row, or column layouts.
- The navigator shows a persistent selected-time cursor with direct manipulation.
- Time-scale presets snap to exact durations around the active selection.
- Processing can be launched from any compatible object and applied with `Enter`.
- Propagation works as a general capability with configurable timestep/frame settings.
- Transform runs emit visible lifecycle feedback.
- Dense stacked-channel views remain readable through grouped axis labeling.

### Notes

- This should follow M9 conceptually, but parts of Batch 1 could be executed earlier if the current branch needs immediate UX stabilization.
- The strongest design move here is not any single widget; it is the shift from "views of one selected tensor" to "a workspace of related scientific objects."
